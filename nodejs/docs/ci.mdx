## 簡介

Playwright 測試可以在 CI 環境中執行。我們已為常見的 CI 提供者建立了範例設定。

在 CI 上執行測試的三個步驟：
1. **確保 CI 代理可以執行瀏覽器**：在 Linux 代理中使用[我們的 Docker 映像](./docker.mdx)，或使用 [CLI](./browsers#install-system-dependencies) 安裝您的相依性。
1. **安裝 Playwright**：
   
   ```bash
   # 安裝 NPM 套件
   npm ci
   
   # 安裝 Playwright 瀏覽器和相依性
   npx playwright install --with-deps
   ```
   
1. **執行您的測試**：
   
   ```bash
   npx playwright test
   ```
   
## Workers

我們建議在 CI 環境中將 [workers](./api/class-testconfig.mdx#test-config-workers) 設為 "1"，以優先考慮穩定性和可重複性。依序執行測試可確保每個測試都能獲得完整的系統資源，避免潛在的衝突。但是，如果您有強大的自託管 CI 系統，您可以啟用[平行](./test-parallel.mdx)測試。若要進行更廣泛的平行化，請考慮使用[分片](./test-parallel.mdx#shard-tests-between-multiple-machines) - 將測試分散到多個 CI 工作中。

```js title="playwright.config.ts"
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  // 在 CI 上選擇退出平行測試。
  workers: process.env.CI ? 1 : undefined,
});
```

## CI 設定

[命令列工具](./browsers#install-system-dependencies)可用於在 CI 中安裝所有作業系統相依性。

### GitHub Actions

#### 在 push/pull_request 時

測試將在主要/主分支的推送或 pull request 時執行。[workflow](https://docs.github.com/en/actions/using-workflows/about-workflows) 將安裝所有相依性、安裝 Playwright，然後執行測試。它還會建立 HTML 報告。

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright Browsers
      run: npx playwright install --with-deps
    - name: Run Playwright tests
      run: npx playwright test
    - uses: actions/upload-artifact@v4
      if: ${{ !cancelled() }}
      with:
        name: playwright-report
        path: playwright-report/
        retention-days: 30
```

#### 在 push/pull_request 時（分片）

GitHub Actions 支援[在多個工作之間分片測試](https://docs.github.com/en/actions/using-jobs/using-a-matrix-for-your-jobs)。查看我們的[分片文件](./test-sharding)以了解更多關於分片的資訊，並查看[GitHub Actions 範例](./test-sharding.mdx#github-actions-example)，了解如何設定工作在多台機器上執行測試以及如何合併 HTML 報告。

#### 透過容器

GitHub Actions 支援使用 [`jobs.<job_id>.container`](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idcontainer) 選項[在容器中執行工作](https://docs.github.com/en/actions/using-jobs/running-jobs-in-a-container)。這對於不汙染主機環境的相依性以及在不同作業系統上為例如螢幕截圖/視覺回歸測試提供一致的環境非常有用。

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  playwright:
    name: 'Playwright Tests'
    runs-on: ubuntu-latest
    container:
      image: mcr.microsoft.com/playwright:v1.55.0-noble
      options: --user 1001
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: lts/*
      - name: Install dependencies
        run: npm ci
      - name: Run your tests
        run: npx playwright test
```

#### 在部署時

這將在 [GitHub Deployment](https://developer.github.com/v3/repos/deployments/) 進入 `success` 狀態後開始測試。像 Vercel 這樣的服務使用這種模式，因此您可以在他們部署的環境上執行端對端測試。

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  deployment_status:
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    if: github.event.deployment_status.state == 'success'
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright
      run: npx playwright install --with-deps
    - name: Run Playwright tests
      run: npx playwright test
      env:
        PLAYWRIGHT_TEST_BASE_URL: ${{ github.event.deployment_status.target_url }}
```

#### Fail-Fast

大型測試套件可能需要很長時間才能執行。透過使用 `--only-changed` 標誌執行初步測試執行，您可以首先執行可能失敗的測試檔案。這將為您提供更快的回饋迴圈，並在處理 Pull Requests 時稍微降低 CI 消耗。為了偵測受您變更集影響的測試檔案，`--only-changed` 會分析您套件的相依性圖表。這是一個啟發式方法，可能會遺漏測試，因此在初步測試執行後總是執行完整的測試套件很重要。

```yml title=".github/workflows/playwright.yml" {24-26}
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        # Force a non-shallow checkout, so that we can reference $GITHUB_BASE_REF.
        # See https://github.com/actions/checkout for more details.
        fetch-depth: 0
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright Browsers
      run: npx playwright install --with-deps
    - name: Run changed Playwright tests
      run: npx playwright test --only-changed=$GITHUB_BASE_REF
      if: github.event_name == 'pull_request'
    - name: Run Playwright tests
      run: npx playwright test
    - uses: actions/upload-artifact@v4
      if: ${{ !cancelled() }}
      with:
        name: playwright-report
        path: playwright-report/
        retention-days: 30
```

### Azure Pipelines

對於 Windows 或 macOS 代理，不需要額外的設定，只需安裝 Playwright 並執行您的測試。

對於 Linux 代理，您可以使用[我們的 Docker 容器](./docker.mdx)搭配 Azure Pipelines 支援[執行容器化工作](https://docs.microsoft.com/en-us/azure/devops/pipelines/process/container-phases?view=azure-devops)。或者，您可以使用[命令列工具](./browsers#install-system-dependencies)來安裝所有必要的相依性。

要執行 Playwright 測試，請使用此管線任務：

```yml
trigger:
- main

pool:
  vmImage: ubuntu-latest

steps:
- task: NodeTool@0
  inputs:
    versionSpec: '18'
  displayName: 'Install Node.js'
- script: npm ci
  displayName: 'npm ci'
- script: npx playwright install --with-deps
  displayName: 'Install Playwright browsers'
- script: npx playwright test
  displayName: 'Run Playwright tests'
  env:
    CI: 'true'
```

#### 使用 Azure Pipelines 上傳 playwright-report 資料夾

如果任何 Playwright 測試失敗，這將使管線執行失敗。如果您還想將測試結果與 Azure DevOps 整合，請像這樣使用 `PublishTestResults` 任務：

```yml
trigger:
- main

pool:
  vmImage: ubuntu-latest

steps:
- task: NodeTool@0
  inputs:
    versionSpec: '18'
  displayName: 'Install Node.js'

- script: npm ci
  displayName: 'npm ci'
- script: npx playwright install --with-deps
  displayName: 'Install Playwright browsers'
- script: npx playwright test
  displayName: 'Run Playwright tests'
  env:
    CI: 'true'
- task: PublishTestResults@2
  displayName: 'Publish test results'
  inputs:
    searchFolder: 'test-results'
    testResultsFormat: 'JUnit'
    testResultsFiles: 'e2e-junit-results.xml'
    mergeTestResults: true
    failTaskOnFailedTests: true
    testRunTitle: 'My End-To-End Tests'
  condition: succeededOrFailed()
- task: PublishPipelineArtifact@1
  inputs:
    targetPath: playwright-report
    artifact: playwright-report
    publishLocation: 'pipeline'
  condition: succeededOrFailed()

```

注意：需要透過以下方式相應地設定 JUnit 報告器

```js
import { defineConfig } from '@playwright/test';

export default defineConfig({
  reporter: [['junit', { outputFile: 'test-results/e2e-junit-results.xml' }]],
});
```

在 `playwright.config.ts` 中。

### CircleCI

在 CircleCI 上執行 Playwright 與在 GitHub Actions 上執行非常相似。為了指定預建的 Playwright [Docker 映像](./docker.mdx)，只需在您的設定中使用 `docker:` 修改代理定義，如下所示：

```yml
executors:
  pw-noble-development:
    docker:
      - image: mcr.microsoft.com/playwright:v1.55.0-noble
```

注意：使用 docker 代理定義時，您將 playwright 執行的資源類別指定為 'medium' 級別[這裡](https://circleci.com/docs/configuration-reference?#docker-execution-environment)。Playwright 的預設行為是將 workers 數量設定為偵測到的核心數（在 medium 級別的情況下為 2）。將 workers 數量覆寫為大於此數量將導致不必要的逾時和失敗。

#### 在 CircleCI 中分片

CircleCI 中的分片以 0 為索引，這意味著您需要覆寫預設的並行性環境變數。以下範例示範如何透過將 1 加到 `CIRCLE_NODE_INDEX` 來使用 CircleCI 並行性 4 執行 Playwright，以傳遞到 `--shard` CLI 參數。

```yml
  playwright-job-name:
    executor: pw-noble-development
    parallelism: 4
    steps:
      - run: SHARD="$((${CIRCLE_NODE_INDEX}+1))"; npx playwright test --shard=${SHARD}/${CIRCLE_NODE_TOTAL}
```

#### 分片

GitLab CI 支援使用 [parallel](https://docs.gitlab.com/ee/ci/yaml/index.html#parallel) 關鍵字[在多個工作之間分片測試](https://docs.gitlab.com/ee/ci/jobs/job_control.html#parallelize-large-jobs)。測試工作將被分割成多個平行執行的較小工作。平行工作從 `job_name 1/N` 到 `job_name N/N` 依序命名。

```yml
stages:
  - test

tests:
  stage: test
  image: mcr.microsoft.com/playwright:v1.55.0-noble
  parallel: 7
  script:
    - npm ci
    - npx playwright test --shard=$CI_NODE_INDEX/$CI_NODE_TOTAL
```

GitLab CI 也支援使用 [parallel:matrix](https://docs.gitlab.com/ee/ci/yaml/index.html#parallelmatrix) 選項在多個工作之間分片測試。測試工作將在單一管線中平行執行多次，但每個工作實例使用不同的變數值。在下面的範例中，我們有 2 個 `PROJECT` 值和 10 個 `SHARD` 值，總共要執行 20 個工作。

```yml
stages:
  - test

tests:
  stage: test
  image: mcr.microsoft.com/playwright:v1.55.0-noble
  parallel:
    matrix:
      - PROJECT: ['chromium', 'webkit']
        SHARD: ['1/10', '2/10', '3/10', '4/10', '5/10', '6/10', '7/10', '8/10', '9/10', '10/10']
  script:
    - npm ci
    - npx playwright test --project=$PROJECT --shard=$SHARD
```

## 快取瀏覽器

不建議快取瀏覽器二進位檔，因為還原快取所需的時間與下載二進位檔所需的時間相當。特別是在 Linux 下，需要安裝[作業系統相依性](./browsers.mdx#install-system-dependencies)，這些是無法快取的。

如果您仍然想在 CI 執行之間快取瀏覽器二進位檔，請在您的 CI 設定中快取[這些目錄](./browsers.mdx#managing-browser-binaries)，針對 Playwright 版本的雜湊值。

## 偵錯瀏覽器啟動

Playwright 支援 `DEBUG` 環境變數在執行期間輸出偵錯日誌。將其設定為 `pw:browser` 在偵錯 `Error: Failed to launch browser` 錯誤時很有幫助。

```bash
DEBUG=pw:browser npx playwright test
```

## 執行有頭瀏覽器 (Running headed)

預設情況下，Playwright 以無頭模式啟動瀏覽器。請參閱我們的[執行測試](./running-tests.mdx#run-tests-in-headed-mode)指南，了解如何在有頭模式下執行測試。

在 Linux 代理上，有頭執行需要安裝 [Xvfb](https://en.wikipedia.org/wiki/Xvfb)。我們的 [Docker 映像](./docker.mdx)和 GitHub Action 都預先安裝了 Xvfb。要使用 Xvfb 在有頭模式下執行瀏覽器，請在實際命令前加上 `xvfb-run`。

```bash
xvfb-run npx playwright test
```
